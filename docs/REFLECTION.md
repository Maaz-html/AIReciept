# Project Reflection — SpendSmart AI

## 1. What was the most challenging technical hurdle?
The most significant hurdle hit me on Day 2 when the `/api/audit/[id]` dynamic routes started returning consistent 404 errors during testing. I spent hours debugging the database connection and route parameters, eventually realizing that the unique IDs weren't being passed correctly between the client-side redirect and the server-side fetch. This was a massive bottleneck that stalled the core user flow. Just as I thought I had stabilized the project, I tried to push my progress to GitHub, only to have the CI/CD pipeline fail with a wall of linting and TypeScript errors. It felt like I was fighting the tools rather than building the product. 

Mid-project, I got over-ambitious and tried to redesign the entire UI to look more "premium," but the new styles broke my form's state management, turning a minor tweak into a 24-hour stabilization phase. The final "facepalm" moment came during production testing: my email redirection was failing because I had hardcoded `localhost:3000` in my Resend template. I had to learn the hard way about environment variables and deployment-specific testing. Overcoming these issues required a lot of trial and error, but it taught me that development isn't just about writing code—it's about managing the friction between different systems.

## 2. If you had 2 more weeks, what feature would you add?
If I were granted another two weeks, my absolute priority would be implementing a CSV/Invoice upload feature. Right now, users have to manually input their data tool-by-tool, which is fine for small teams but becomes a high-friction task for larger startups. I would want to allow users to drag and drop their billing exports from OpenAI or Anthropic directly into the app, using a parser to automatically populate the audit fields. This would move the app from a simple "calculator" to a true "automated audit platform," which is much more valuable for the busy operations managers we are targeting.

Additionally, I would build a robust "Monthly Spend Tracking" dashboard that allows users to visualize their savings trends over time, rather than just seeing a one-off snapshot. Finally, I would implement a "Professional PDF Export" engine. Founders often need to present these findings to their boards or finance teams; having a well-formatted, Credex-branded PDF report would make the savings feel more official and shareable. These features would collectively turn a quick web tool into a persistent financial utility that teams could return to every month as their AI stack evolves.

## 3. Which architectural decision paid off the most?
The decision that saved the project was choosing a deterministic, "auditing-level" calculation engine over a purely AI-driven one. Early on, I was tempted to just send all the user's raw inputs to an LLM and ask it to "figure out the savings," but I quickly realized that AI hallucinations in financial math would destroy user trust. Instead, I built a hardcoded logic layer in `lib/auditEngine.ts` that uses verified pricing data to calculate precise savings. This was much more efficient, required zero API latency for the core math, and ensured 100% accuracy for every audit.

This architectural choice paid off immensely during testing. I could write unit tests for specific rules—like recommending Cursor Pro when a team is overpaying for separate Claude seats—and know they would work exactly the same way every time. We still used AI strategically to generate the "AI Summary" and qualitative insights, which is where LLMs actually shine, but keeping the "math" deterministic gave the app a professional backbone. It taught me that while AI is a powerful tool, it’s often the "boring" code—the well-structured logic and data—that provides the most reliability in a production environment.

## 4. What was the biggest "lesson learned" during the build?
The biggest lesson I’m taking away from this project is the delicate balance between having a rigid plan and being willing to pivot. I started with a very structured roadmap, but as soon as the UI broke and the API errors started piling up, I realized that a developer's real job is troubleshooting, not just typing. I learned that it is essential to have a plan to keep from getting lost, but you have to be flexible enough to change that plan the second the data shows it's not working. You can't be so in love with your initial design that you refuse to fix it when it ruins the user experience.

The other major takeaway was the importance of "continuous testing." I used to think testing was a final step you did at the end of a sprint, but this project proved that you need to test as you go. Catching the `localhost` URL bug earlier would have saved me a lot of stress during the deployment phase. I’ve come out of this build understanding that a "finished" feature isn't just code that runs on my machine; it’s code that is linted, tested, and verified in the environment where it will actually live. It’s changed how I think about "completion" as a developer.

## 5. How did your understanding of the user change?
Going into this, I assumed that users would want a highly granular, academic breakdown of every single pricing tier and a detailed comparison of why one tool was chosen over another. I spent a lot of time on the tool-by-tool breakdown cards. However, after looking at the results and testing the flow, I realized that users care far less about individual seat prices and far more about the "Total Potential Annual Savings" number in the hero section. That is the one number that determines whether they find the tool valuable or not.

Most of our target users are busy Engineering Managers or Founders who are drowning in subscriptions. They aren't looking for another complex dashboard to manage; they are looking for a quick, "Aha!" moment that tells them exactly how much money they can recover. This realization changed my design priority from "transparency of data" to "clarity of outcome." I learned that in a lead-generation tool like this, you have to lead with the biggest value proposition immediately. My understanding shifted from building a "data tool" to building an "insight tool"—the user wants to know the *result* of the audit, not necessarily the math behind it.
